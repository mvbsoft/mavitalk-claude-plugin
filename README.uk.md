# mavitalk — пайплайн сесій для Claude Code

> 🇬🇧 English: [README.md](README.md)

`mavitalk` — це особистий **плагін для Claude Code** (у маркетплейсі `mavitalk-claude-plugin`),
який обгортає твої сесії кодування в дисциплінований, майже безручний процес:

- **У кінці сесії** він перевіряє роботу (багатоагентне рев'ю за рівнями), допомагає виправити
  знайдене, робить **чистий gated-коміт** і пише повний хендоф.
- **На початку наступної сесії** він відновлює все — що зроблено, що лишилось, який точний наступний
  крок — щоб ти продовжив, нічого не пояснюючи заново.

Уся його пам'ять лежить в одній теці — **`.superhelpers/`**, англійською; розмова ж триває твоєю
мовою.

## Для чого це

Проблема: закривати сесію «наосліп» (комітити неперевірене, втрачати *чому* за рішеннями) і щоразу
пояснювати контекст знову. mavitalk робить **кінець** сесії строгим, а **старт** наступної —
безшовним; керується короткими природними фразами, а не довгими промптами.

## Два скіли

| Скіл | Тригер (будь-яка мова) | Що робить |
|---|---|---|
| `mavitalk:finishing-the-session` | «давай закінчуємо», «let's finish», «wrap up» | Оцінка → рев'ю за рівнем → фікс → **gated-коміт** → запис хендофу/пам'яті |
| `mavitalk:continue-session` | «продовжуємо», «давай почнемо», «let's continue» | Відновлює контекст минулої сесії, звіряє стан git, продовжує твоєю мовою |

Надійність дають два вбудовані хуки: `UserPromptSubmit` ловить фрази, а `SessionStart` автоматично
вкидає `next-session.md` щойно ти відкриваєш проєкт (контекст підвантажується **без жодного промпта**).

## Як користуватися

**Завершити сесію** — напиши щось на кшталт `давай закінчуємо`. Скіл:
1. Пропонує **рівень перевірки** (фінальне рішення за тобою):

   | Рівень | Що запускає | ≈ вартість |
   |---|---|---|
   | **Light** | 4 рев'ю-агенти (коректність · архітектура · безпека · якість+доки) | ~80k токенів |
   | **Medium** | + Requirement Auditor (чат ↔ код) + Judge | ~110k |
   | **Full** | + детермінований security-скан + ре-рев'ю після фіксу | ~180k |

2. Проганяє рев'ю, допомагає виправити Critical/Important, перезапускає твої гейти.
3. **Зупиняється на гейті коміту** — показує застейджений дифф і чекає твого «ок». Коміти виглядають
   як звичайні dev-коміти (**без згадки AI**). Ніколи не пушить без прохання.
4. Пише хендоф: лог `sessions/`, `project-memory.md` і `next-session.md`.

**Продовжити наступного разу** — відкрий проєкт (контекст підвантажиться сам) і напиши `продовжуємо`.
Воно перечитає хендоф, звірить останній SHA коміту з git (щоб не довіряти застарілому списку «зроблено»),
дасть тобі бриф твоєю мовою і почне найближчу наступну дію.

## Де зберігає все — `.superhelpers/`

```
.superhelpers/
├── sessions/YYYY-MM-DD-NNN.md   # append-only лог сесії          (комітиться)
├── memory/project-memory.md      # rolling пам'ять проєкту         (комітиться)
├── next-session.md               # continuation-контекст           (комітиться)
├── reviews/  staging/            # тимчасовий скретч пайплайну      (gitignore)
└── config.yml                    # гейти, мова, рівні, атрибуція
```

На першому завершенні в проєкті mavitalk сам розгортає цю теку з шаблонів. Налаштуй `config.yml` —
команди гейтів, `review.reviewer_model` (дефолт `sonnet`), `review.default_tier`, `attribution.commit`
(дефолт `none`).

## Встановлення

Поки маркетплейс — локальна тека:
```
/plugin marketplace add ~/projects/mavitalk-claude-plugin
/plugin install mavitalk@mavitalk-claude-plugin
```
Коли запушено в git-хост:
```
/plugin marketplace add <твій-github-юзер>/mavitalk-claude-plugin
/plugin install mavitalk@mavitalk-claude-plugin
```
Після правок: `/plugin marketplace update mavitalk-claude-plugin`, потім `/reload-plugins`.

## Архітектура налаштування — що де живе

Конфіг Claude Code має **три шари**, які поводяться по-різному, коли репо переїжджає на іншу машину
чи до іншого розробника:

| Шар | Лежить у | Їде з `git clone`? | Хто бачить |
|---|---|---|---|
| **Глобально (user)** | `~/.claude/` | ❌ ні — поза будь-яким репо | лише ти, лише ця машина |
| **Плагін (цей маркетплейс)** | репо `mavitalk-claude-plugin` | ✅ через `/plugin install` | усі, хто додав маркетплейс |
| **Проект (закомічене)** | `.claude/` + `.mcp.json` кожного репо | ✅ так | усі, хто клонує репо |

Правило:

- **Глобально `~/.claude/`** — мінімум: тільки персональне й машинне (твій приватний `CLAUDE.md`,
  тема/модель/statusline, auth-токени MCP). Усе, що має бути відтворюваним, **не** може жити лише тут.
- **Цей плагін/маркетплейс** — спільний, відтворюваний тулсет: воркфлоу `mavitalk`, куровані
  сторонні плагіни-стандарти, спільні MCP-сервери. Єдине джерело правди для всіх репо й машин.
  Тримай його **project-agnostic** (без специфіки жодного окремого репо).
- **Закомічений `.claude/` кожного репо** — суто проектне: проектні скіли, команди, агенти, хуки,
  дозволи, проектні MCP. Це вже їде з кодом.

### Один install = увесь тулсет

Сторонні плагіни ти **не копіюєш** (vendoring) сюди. Чотири нативні механізми (усі перевірені по
документації Claude Code) дають «одну дію → все»:

1. **Агрегація маркетплейсу** — запис плагіна в `marketplace.json` може мати `source` на *зовнішнє*
   репо (`github` / `url` / `git-subdir` / `npm`), тож один маркетплейс ре-лістить сторонні плагіни
   (superpowers, security-audit, …), які лишаються під своїм upstream.
2. **Залежності плагінів** — `plugin.json` має масив `dependencies` (semver). Встановлення / увімкнення
   плагіна авто-ставить **і вмикає** оголошені залежності — один плагін тягне весь набір при install.
   Залежності вмикаються **на тому ж scope**, що й батьківський: увімкнув user-wide → і вони user-wide;
   запінив на один проєкт → лишаються project-scope.
3. **Project-pin** — закомічені `extraKnownMarketplaces` + `enabledPlugins` у `.claude/settings.json`
   репо змушують Claude Code запропонувати поставити весь набір, коли розробник клонує й довіряє репо
   (офіційний «team» механізм).
4. **MCP усередині плагіна** — плагін може везти MCP-сервери (`.mcp.json` / `mcpServers` у
   `plugin.json`). Секрети під кожного розробника — через `userConfig` (`sensitive: true`,
   `${user_config.KEY}`), тож кожен вписує свій.

### Оновлення — чесна правда

Вкладеного «плагіна в плагіні, що авто-оновлює дітей», **немає**. Кожен плагін оновлюється зі **свого
джерела**:

- Агреговані / залежні плагіни оновлюються зі своїх upstream-репо — ти отримуєш їхні фікси
  автоматично, а не через оновлення цього плагіна.
- `dependencies` авто-*ставляться* при ввімкненні, але не форсують *оновлення* при апдейті цього
  плагіна.
- **Vendoring** (копія чужого коду сюди) — єдиний спосіб «оновив це = оновив усе», але це форк і
  втрата upstream-фіксів; уникай для живих плагінів.

Тож тримай сторонні плагіни **referenced, не copied**; `/plugin marketplace update` оновлює, а кожен
плагін тягне свій upstream.

## Варто знати

- **Контроль лишається за тобою.** Коміт гейтований; нічого не пушиться без тебе. Пайплайн *пропонує*
  рівень — рішення твоє.
- **Це порада, а не клітка.** Claude виконує інструкції скілів; хуки роблять тригер і відновлення
  контексту надійними, але глибина конкретного прогону — це поведінка Claude + твої підтвердження.
- **Економія токенів:** на дрібних сесіях — Light/skip; Full лишай для суттєвої роботи.

## Розробка

Скіли й хуки — під `plugins/mavitalk/`. Редагуй тут (не копію в `~/.claude/plugins/cache`),
проганяй shell-тести (`sh plugins/mavitalk/tests/run-tests.sh`), потім
`/plugin marketplace update` + `/reload-plugins`.
