# Життєвий цикл налаштувань плагіна

Специфікація того, як плагін знаходить, створює, перевіряє й залежить від файлу
`.mavitalk/config.yml` проєкту. Супутник до `agent-fanout-governor.md`.

## Проблема

Функції сесійного циклу (ворота якості, багаторівневе рев'ю, `end-session`) потребують кількох
проєктно-специфічних фактів — насамперед команд, що запускають тести/лінт/типи/формат. Зараз ці факти
живуть в опційному `.mavitalk/config.yml`, який читає модель. Звідси три проблеми:

- **Мовчазне псування.** Застарілий чи хибний конфіг (мертві ключі, неправильний `paths.root`) погіршує
  поведінку без жодного сигналу — напр. забутий ліміт тихо стискає хвилю рев'ю.
- **Немає керованого налаштування.** Новий проєкт не має ні конфіга, ні допомоги його скласти; плагін
  падає на вгадування стеку, що ненадійно для контейнерних репозиторіїв.
- **Немає контракту активації.** Плагін виконує свій цикл навіть тоді, коли йому нема чого надійно
  запускати, замість того щоб стати в сплячку, доки його не налаштують.

## Цілі

- Файл конфіга **обов'язковий** для роботи функцій сесійного циклу; його відсутність присипляє ці
  функції замість вгадування.
- **Керований, напівавтоматичний** майстер сканує проєкт, пропонує налаштування, пояснює кожне й пише
  файл лише після підтвердження розробником.
- При старті сесії плагін **перевіряє** конфіг і одразу показує проблеми, пропонуючи виправити.
- Поведінка **безпечна без нагляду**: headless-сесія без валідного конфіга лишається у сплячці й ніколи
  не самоналаштовується та не пише.

## Поза межами

- Заміна проєктних файлів-інструкцій (`AGENTS.md`). Вони лишаються людськими правилами проєкту; конфіг
  тримає лише машинні факти, які споживає цикл.
- Рантайм-залежність. Хуки лишаються чистим `bash`; жоден вбудований парсер/інтерпретатор не додається.
- Зміна крос-проєктних інженерних стандартів — вони не проєктно-специфічні.

## Ключові рішення

| Рішення | Вибір |
|---|---|
| Обсяг сплячки | **Стандарти діють завжди; у сплячку йде лише проєктно-специфічний сесійний цикл** за відсутності конфіга. |
| Суворість валідації | **Два рівні.** Структурна поломка (не парситься / ключі не тієї форми) — 🔴 блокер, який тримає цикл у сплячці, доки не полагоджено; порадні проблеми (не знайдено тестів, застарілий/невідомий ключ, лише дефолти) — 🟡 попередження, які підсвічуються, але не блокують. |
| Де живе логіка | **Тонкий детермінований хук + товсті скіли.** Хук робить швидку мілку перевірку й вкидає вердикт; сканування, пропозиція, глибока перевірка й лагодження живуть у скілах. |
| Пошук тестів | `config.yml gates:` → інакше канонічний ранер з `AGENTS.md` → інакше **пропустити тести з гучним попередженням**. |
| Модель активації | Конфіг обов'язковий для циклу, але вимагається лише **в інтерактиві**. Headless + не налаштовано = сплячка + тиша. |
| Механізм «не активний» | **Самонакладена сплячка** (хук вкидає директиву сплячки; скіли самі перевіряють наявність конфіга). У Claude Code немає посесійного рубильника плагіна на рівні рушія; це — практичний еквівалент. |

## Архітектура (тонкий хук + товсті скіли)

Шість компонентів:

1. **Схема + доки конфіга** — єдине джерело правди: кожне налаштування, його дефолт, обов'язкове чи
   опційне, і правила блокер/попередження + автовиправлення. На неї спираються майстер, лікар і README.
   Розширює наявний `templates/mavitalk/config.yml`.
2. **Вартовий при старті (хук)** — розширює наявний SessionStart-хук. Чистий `bash`. Завжди вкидає
   крос-проєктні стандарти. Визначає «людина присутня / headless». Мілко перевіряє `config.yml`
   (є / структурно зламаний / немає) плюс кілька дешевих `grep`-порад. Вкидає директиву-вердикт; сам
   ніколи не питає (хук не вміє) — за директивою діє модель.
3. **Скіл `/mavitalk:configure`** — майстер і точка входу для ремонту. Сканує проєкт, пропонує конфіг,
   проводить розробника по кожному налаштуванню з простим поясненням, пише `config.yml` після
   підтвердження. Повторний запуск на наявному файлі — лагодить його.
4. **Лікар-валідатор (skill-reference)** — одна спільна копія правил глибокої перевірки: схема, типи,
   невідомі/застарілі ключі, чи взагалі є звідки взяти тести, поділ на блокер/попередження й межа
   «саме vs із підтвердженням». Застосовується і старт-фіксом, і майстром. Не окремий агент.
5. **Правка пошуку тестів** — `end-session` читає ворота за ланцюжком вище; текст `SKILL.md` і
   `verification-rubric.md` оновлено відповідно.
6. **Тести** — `bash`-набір плагіна, розширений під нову поведінку; наприкінці весь набір зелений.

Межі: хук не приймає семантичних рішень (лише є / зламано / немає × людина / headless); уся семантика — у
скілах. Лікар — одна копія-reference, а не три копії «що таке валідно».

## Схема конфіга

Розпізнані налаштування (усі з дефолтами) групуються так: `language`, `attribution`, `gates`, `review`
(тір, моделі, ростери, активація, пороги), `throttle`, `security`, `project`, `paths`. Доки перелічують
**кожен** ключ, який реально читають скіли.

- **Мінімум, щоб вважатися налаштованим:** коректно сформований файл із розпізнаними верхніми секціями й
  без структурного псування. Жоден окремий ключ не обов'язковий — у кожного є дефолт, а `gates:` можна
  делегувати в `AGENTS.md`. Наявність + валідна структура і є сигналом активації (запуск майстра — це і є
  явна згода).
- **🔴 Блокер:** YAML не парситься; наявний ключ не тієї форми, через що цикл зламається (`gates` не
  словник, ростер не список, `throttle.hard_cap` не число).
- **🟡 Попередження:** ніде не знайдено команд тестів (конфіг порожній і немає ранера в `AGENTS.md`) →
  тести пропускатимуться; застарілий/невідомий ключ (напр. `max_review_agents`); файл лише з дефолтів;
  `paths.root`, що не збігається з реальним місцем артефактів.
- **Автовиправлення (без запиту):** викинути застарілі мертві ключі, полагодити `paths.root`,
  нормалізувати форматування. **Спершу підтвердити:** усе, що змінює поведінку — команди тестів, моделі,
  тіри, ростери, активація, атрибуція.

## Потоки

**A — старт сесії (вартовий).**
1. Вкинути крос-проєктні стандарти (завжди).
2. Визначити «людина присутня / headless».
3. Класифікувати `config.yml`: **OK** / **НЕМАЄ** / 🔴**БЛОКЕР**, плюс дешеві `grep`-поради.
4. Вкинути вердикт:
   - OK → цикл озброєний, мовчки.
   - НЕМАЄ / БЛОКЕР + людина → «цикл у сплячці: `<причина>`; запропонуй `/mavitalk:configure`».
   - НЕМАЄ / БЛОКЕР + headless → сплячка, мовчки, нічого не писати.
5. В інтерактиві: модель раз називає причину й пропонує майстер. Відмова → сплячка на сесію, без
   повторних нагадувань.

**B — майстер (`/mavitalk:configure`).** Читальне сканування (стек, Docker/compose, цілі `Makefile`,
проза воріт в `AGENTS.md`, наявний конфіг) → чернетка з дефолтів + знайдених воріт (ворота
**пропонуються, ніколи не вгадуються мовчки**) → провести по кожному налаштуванню з поясненням і
підтвердженням/правкою → записати `.mavitalk/config.yml` (створити `.mavitalk/`, якщо немає; ніколи не
чіпати `sessions/` чи `memory/`) → підсумок; цикл озброєний.

**C — ремонт (лікар).** Спрацьовує, коли наявний конфіг позначено, або майстер запущено на наявному
файлі: завантажити → перелічити 🔴 блокери + 🟡 попередження простими словами → автоматично застосувати
косметичні/застарілі фікси; зміни, що впливають на поведінку, — запропонувати з підтвердженням → знову
перевірити → відзвітувати.

## Обробка помилок і межові випадки

- YAML не парситься → блокер → сплячка + пропозиція ремонту (людина) / тиша (headless).
- `.mavitalk/` є без `config.yml` → НЕМАЄ; теки стану (`sessions/`, `memory/`) не чіпаються.
- headless + немає/поганий → сплячка, тиша, ніколи не пише. Лише env-override власника міг би це зняти,
  як у `agent-throttle`.
- Розробник відмовився від налаштування → сплячка на сесію; не перепитувати.
- Валідний конфіг, але тестів нема ніде → попередження; `end-session` працює, але пропускає тести гучно.
- Невпевненість при автофіксі → питати; ніколи не переписувати мовчки ключ, що впливає на поведінку.

## План тестів

- **Класифікація вартового:** фікстури немає / структурно зламано / валідно × людина / headless →
  перевірити вкинуту директиву.
- **Grep-поради:** застарілий ключ → порада; тестів нема ніде → порада.
- **Лікар:** класифікація блокер vs попередження на зразках; межа «саме vs із підтвердженням».
- **Пошук тестів:** конфіг-ворота беруться; порожній конфіг + ранер в `AGENTS.md` → з `AGENTS.md`; ніде →
  пропуск + попередження.
- **Вивід майстра:** зі сканованої фікстури записаний конфіг відповідає схемі.
- **Схема/доки:** шаблонний `config.yml` валідний за схемою; доки перелічують кожен ключ, який читають
  скіли.
- Наприкінці весь `tests/run-tests.sh` зелений.

## Відхилені варіанти

- **Повна сплячка** (не вкидати нічого без конфіга, разом зі стандартами). Відхилено: крос-проєктні
  стандарти не проєктно-специфічні, тож обумовлювати їх наявністю проєктного файлу — карати не за те.
- **Товстий хук** (вбудований парсер робить повну валідацію детерміновано). Відхилено: додає
  рантайм-залежність і ламає нульову портативність плагіна, а також суперечить правилу «усе в контейнері»,
  яке тримає тулчейни поза хостом.
- **Тільки скіл / ліниво** (без перевірки при старті; вимагати на першому `end-session`). Відхилено: не
  виконує вимогу перевіряти й пропонувати налаштування при старті й губить привітання на вході.

## Викочування

Постачається як одна функція з тестами. Природний внутрішній порядок збірки — спершу схема/доки +
вартовий (щоб існував контракт активації), тоді майстер і лікар — лишається плану впровадження.
